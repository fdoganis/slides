# [Self documenting code](https://stackoverflow.com/a/209089)
- document the "why?"
  - at high-level
  - if needed
- avoid documentation altogether
  - write tests, examples, tutorials instead
  - write **code as documentation**

# :point_down:

# Code as documentation

- **BAD** :x:

```cpp
const float a = 9.81; //gravitational force
float b = 5; //time in seconds
float c = (1/2)*a*(b^2) //multiply the time and gravity together to get displacement.
  ```

- **GOOD** :white_check_mark:
 ```cpp
float computeDisplacement(float timeInSeconds) {
    const float gravitationalForce = 9.81;
    float displacement = (1 / 2) * gravitationalForce * (timeInSeconds ^ 2);
    return displacement;
}
```

# [Naming Conventions](https://medium.com/swlh/naming-conventions-101-for-developers-8997bb96fd60)
- "What is this object? What does it do?"
- Object, variable :arrow_right: noun
  - `user`, `accountNumber`, `customerEmail`
- Function, method :arrow_right: verb
  - `user.login()`, `shutDown()`
- Boolean :arrow_right: adjective
  - `allowed`, `disabled`
  -  `user.active()` :x:
  - `user.isActive()` :white_check_mark:


---

# [Naming tips](https://hackernoon.com/tips-for-writing-maintainable-code-naming-things)

- convey **intention**
`var d // elapsed time in days` :x:  vs `var elapsedTimeInDays` :white_check_mark:
- name **arguments** too!
`void copyChars(char a1[], char a2[])` :x:
`void copyChars(char source[], char destination[])` :white_check_mark:
- **pronunciation** (avoid abbreviations)
`genymdms` :x: vs `generationTimestamp` :white_check_mark:
- **no magic numbers** (keep your code searchable and maintainable)
`s / 5` :x: vs `task / WORK_DAYS_PER_WEEK` :white_check_mark:
- use **camelCase**
